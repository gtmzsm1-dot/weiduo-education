## 决策 #1:用户与权限模型(2026-05-09 确立)

### 用户层级

| 层级 | 角色 | 描述 |
|---|---|---|
| **L1** | 校长(大股东) | 系统 owner,拥有最高权限和所有数据组合视角 |
| **L2** | 部门主管(小股东) | 参与运营,身份是合伙人 + 主管 + 员工的复合 |
| **L3** | 普通老师 | 教学执行者,只关心自己学生 |
| **L4** | 家长 | 通过档案码查看自己孩子的档案 |
| **L5** | 学生 | (暂未实现) |

### 核心设计原则:组合视角分级

**不采用**"硬字段分级"(主管看不到某些字段)。
**采用**"组合视角分级"(字段尽量开放,L1 的护城河在数据的组合方式、聚合维度、对比视角、决策上下文)。

#### 实操规则

1. **单点开放,聚合独享**:
   - 主管能看到自己业务条线的单个学员/老师/家长信息
   - 跨条线对比、全机构聚合、横向排行 → L1 独享

2. **看一下 ≠ 带走**:
   - 主管能逐个查看自己条线的家长联系方式(用于联系)
   - 批量导出、全量下载、跨条线导出 → L1 独享

3. **数据 ≠ 解读**:
   - 主管能看到自己条线的原始数据
   - 数据的组合解读、趋势预警、异常识别 → L1 独享(通过 L1 驾驶舱呈现)

#### 该原则带来的好处

- 主管不感觉"被屏蔽某些字段",对系统信任度高
- L1 的决策空间通过"独享视角"保护,而非"独享数据"
- 主管即使离职/流失,带走的是单点数据,不是机构经营全貌
- 减少内部权限引发的人际摩擦

### 当前实现差距

当前系统(2026-05-09)只有 L1(管理员密码)和 L4(家长档案码),L2/L3 尚未实现。

完整的多级权限实现必须等阶段 7 接入后端之后(纯前端架构无法做出可信的权限分级)。**在此之前,任何"看似分级"的设计都不可信**,不应该启动。

### 阶段 6 之前可以做的准备工作

1. **L1 独享驾驶舱**:在系统中加一个隐藏入口(如 `?super=1` + 独立密码),集中呈现"组合视角"内容(全机构 KPI、对比维度、私人备注等)。这部分内容**主管即使知道入口也进不去**。
2. **数据分级清单**:由 owner 独立维护一份"L1 独享视角清单",定义哪些组合方式属于 L1 独享。这份清单是后续后端权限设计的依据。

### 关联决策点

- 阶段 2.4 家长档案码改造:必须考虑"档案码不能反推出主管视角"
- 阶段 7 架构评估:后端方案选型需评估"分级权限 + 防批量导出"的实现成本
- 任何新功能在设计阶段必须回答:"L1 看到什么?L2 看到什么?是字段差异还是组合差异?"

## 决策 #2:产品成功标准(2026-05-09 确立)

### 系统目标

**让一家机构在 3-5 年后,比"没有这个系统"的状态,综合体验和经营结果更好**。

### 不追求的指标

- 功能数量
- 数据维度的丰富度
- 录入字段的全面性

### 真正追求的体验

- 老师感觉"系统在帮我",而不是"系统在监控我"
- 家长感觉"这家机构真的在认真带我孩子"
- L1 感觉"我每天 5-10 分钟就能掌握全局"
- 数据真实(优于数据全面)

### 反直觉原则:克制比加法重要

自用系统更容易越做越臃肿。每季度评估一次:

- 哪些功能 30 天没人用?→ 强制下线
- 哪些功能录入率 < 50%?→ 重新设计或下线
- 哪些功能让老师每次操作多 3 步以上?→ 重做交互

### 关联决策点

- 任何新功能提案在评估阶段必须回答:"这个功能让 L1/老师/家长哪一方更轻松了?"
- 阶段 6 用户体验优化的核心目标是"减少老师摩擦",不是"增加功能"

## 决策 #3:产品架构定位(2026-05-09 确立)

### 单机工具 + 云端只读同步混合架构

owner 实际使用场景:**只在自己一台电脑录入和管理学员档案**。
家长使用场景:**用自己手机扫码,在任何地方查看孩子档案,每周约一次**。

### 架构设计

- **录入端(owner)**:纯单机,数据存 localStorage(当前架构不变)
- **同步层**:owner 主动点击"同步到云端",触发加密 JSON 推送至 GitHub repo
- **查看端(家长)**:扫码访问 GitHub Pages,fetch 加密 JSON,用 archiveCode 解密

### 不采用的方案及理由

- **方案:接入 Supabase / Firebase 等后端服务**——拒绝。理由:超出 0 元 + 几天时间预算,且引入第三方依赖
- **方案:全公开 JSON(明文)**——拒绝。理由:K12 学生数据泄露风险,《个人信息保护法》《未成年人保护法》合规问题
- **方案:每个家长二维码编码完整档案**——拒绝。理由:数据更新需重新生成二维码发家长,运营成本高
- **方案:放弃家长扫码功能**——拒绝。理由:owner 明确"作为机构差异化宣传卖点"

### 关联决策

- 此决策**修订**了原决策 #1 中"完整多级权限必须等阶段 7 接后端"的论断——单机 + 云同步混合架构下,家长扫码功能不需要传统后端
- 此决策**优先级高于**未来任何"接入完整后端"的提议,除非业务场景从单 owner 扩展到多 owner
